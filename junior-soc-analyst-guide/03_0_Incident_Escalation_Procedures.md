# Escalate Incidents Based on Procedures

## 📋 Review Points

- [ ] Understand escalation criteria and procedures
- [ ] Know the escalation chain in your organization
- [ ] Understand when to escalate and when not to
- [ ] Know different escalation paths (severity-based, type-based, etc.)
- [ ] Understand communication requirements for escalation
- [ ] Know incident severity definitions
- [ ] Understand SLAs for different incident types
- [ ] Know documentation requirements for escalation

---

## 🔍 Step-by-Step Guide

### Step 1: Understand Escalation Criteria
1. Review your organization's escalation policy
2. Document the criteria for each incident type
3. Understand severity levels (P1, P2, P3, etc.)
4. Know which teams handle which incident types
5. Understand time-based escalation requirements
6. Learn about business impact vs. technical severity

### Step 2: Classify the Incident Correctly
1. Determine incident category (malware, intrusion, data breach, etc.)
2. Assess initial severity level based on:
   - Number of systems affected
   - Criticality of affected systems
   - Type of data potentially exposed
   - Business impact
   - User impact
3. Document classification reasoning
4. Validate classification with supervisor if unsure

### Step 3: Determine Escalation Need
1. Ask: Does this meet escalation criteria?
2. Check severity against escalation thresholds
3. Verify if incident type requires escalation
4. Consider time of day (after-hours incidents may need faster escalation)
5. Check if customer/client notification is needed
6. Document escalation decision rationale

### Step 4: Prepare Escalation Information
1. Gather all relevant alert data and context
2. Create concise summary of the incident
3. Document timeline of events
4. List systems/users affected
5. Note any immediate actions already taken
6. Include relevant logs and screenshots
7. Document risk assessment
8. Prepare recommendations if applicable

### Step 5: Identify Correct Escalation Path
1. Determine which team should handle the incident:
   - **Incident Response Team**: Active intrusions, data breaches
   - **Malware/AV Team**: Malware detections
   - **Network Security**: Network-based attacks
   - **Application Team**: Application-specific issues
   - **Management**: Customer notification, PR
2. Check escalation matrix for correct contact
3. Verify contact information is current
4. Determine if multiple teams need notification

### Step 6: Perform Escalation
1. Use official escalation channels (ticketing system, phone, etc.)
2. Follow escalation template/format
3. Clearly state:
   - Incident description
   - Severity/priority level
   - Affected systems and users
   - Time-sensitive information
   - Requested action
   - Your contact information
4. Send to appropriate person/team
5. Record escalation details and timestamp

### Step 7: Follow Up and Track
1. Confirm escalation was received
2. Document response and actions taken
3. Monitor incident resolution
4. Provide updates if escalated team requests information
5. Update ticket status appropriately
6. Document lessons learned after resolution

## 🎯 Practice Activities

### Activity 1: Escalation Procedures Review (30 minutes)
1. Review your organization's escalation policy document
2. Create a visual map of escalation chain
3. Document contact information for each escalation path
4. Identify escalation criteria for each incident type
5. Create quick reference card with escalation criteria

### Activity 2: Severity Classification Exercise (45 minutes)
1. Review 10 sample incidents
2. Classify each by severity level
3. Determine escalation need for each
4. Document your reasoning
5. Compare with actual classification and discuss

### Activity 3: Escalation Message Drafting (1 hour)
1. Review examples of well-written escalation messages
2. Draft escalation messages for 5 scenarios
3. Include all required information
4. Practice being concise but complete
5. Have team lead review and provide feedback

### Activity 4: Role-Play Escalation Scenarios (1-2 hours)
1. Partner with colleague or supervisor
2. Practice escalating incidents in real-time
3. Handle various scenarios (resistant audience, unclear info, etc.)
4. Receive feedback on communication
5. Practice managing difficult escalations

### Activity 5: SLA Management Exercise (Ongoing)
1. Learn your organization's response time SLAs
2. Track incidents and their SLA compliance
3. Identify patterns of missed SLAs
4. Understand why some incidents escalate faster
5. Practice escalating with appropriate urgency

### Activity 6: Documentation Review (30 minutes)
1. Review well-documented incident escalations
2. Identify key elements of good documentation
3. Practice writing clear, concise escalation summaries
4. Get feedback on your documentation quality
5. Refine your escalation documentation approach

## ✅ Success Criteria

- Escalate incidents within SLA timeframes
- Provide complete and accurate escalation information
- Use correct escalation paths consistently
- Classify severity accurately
- Escalate appropriate incidents without over-escalating
- Maintain clear communication with escalated teams
- Document escalation decisions and outcomes
- Follow up appropriately on escalated incidents
- Reduce escalation time with experience
